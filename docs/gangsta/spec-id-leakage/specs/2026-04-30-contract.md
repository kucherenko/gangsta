# Contract: Spec Identifier Leakage Prevention

## Objective

Prevent Gangsta-internal spec identifiers (FR-xxx, NFR-xxx, WP-xxx) from appearing in
project-facing artifacts outside `docs/gangsta/`. The confirmed leak pathway runs from the
Contract and Execution Plan — which embed these identifiers — through Worker dispatch, where
no existing instruction prohibits workers from reproducing them in deliverables (source code,
test files, comments, README files). This Heist closes that gap by adding a constitutional
Negative Constraint, prevention guidance at two upstream dispatch points, and a single
blocking enforcement gate in the Laundering verification checklist.

## Requirements

### Functional Requirements

1. FR-001: `docs/gangsta/constitution.md` MUST include a new Negative Constraint (NC-002)
   prohibiting Gangsta-internal spec identifiers (FR-xxx, NFR-xxx, WP-xxx) from appearing in
   project-facing artifacts outside `docs/gangsta/`, including source code, test files, code
   comments, and README files.

2. FR-002: NC-002 MUST be scoped to confirmed-leak identifier types only: FR-xxx, NFR-xxx,
   WP-xxx. AC-xxx and AD-xxx MUST NOT be included in NC-002 or in any scan pattern, as no
   leakage of those types has been observed.

3. FR-003: NC-002 MUST carry a source annotation (`— Source: spec-id-leakage Heist`),
   consistent with the source-annotation standard established by this Heist for all NC entries.
   The source-annotation standard is: append `— Source: <heist-id> Heist` at the end of the
   NC entry text, with no other required fields. Placement is always end-of-line.

4. FR-004: `skills/resource-development/SKILL.md` MUST include prevention guidance in the
   Work Package brief template section, explicitly naming the canonical identifier forms
   (FR-xxx, NFR-xxx, WP-xxx where xxx is one or more digits) and instructing workers not to
   reproduce them in deliverables. This addition is labeled as prevention guidance, not a gate.

5. FR-005: `skills/the-hit/SKILL.md` MUST include prevention guidance in the worker dispatch
   checklist, explicitly naming the canonical identifier forms and instructing workers not to
   reproduce them in source code, tests, comments, or documentation. This addition is labeled
   as prevention guidance, not a gate.

6. FR-006: `skills/laundering/SKILL.md` MUST include an identifier scan as a blocking gate in
   the verification checklist. The gate MUST:
   a. Search all project files outside `docs/gangsta/` — including source code, tests, code
      comments, and README files.
   b. Use the pattern `\b(FR|NFR|WP)-\d+\b` as the canonical match expression.
   c. Treat any match as a blocking defect requiring removal and re-verification before the
      Heist can be declared clean.
   d. Allow documented exceptions: if a match is a legitimate project identifier unrelated to
      Gangsta planning, the agent MUST record the matching string, the file path, and a
      one-sentence justification in the laundering checkpoint for Don review. The Don MUST
      acknowledge the exception as part of the checkpoint review sign-off before the gate is
      cleared.

7. FR-007: NC-001 in `docs/gangsta/constitution.md` no longer carries a source annotation
   (the source Heist directory was removed when autonomous-mode was deprecated). NC-001's text
   was generalized to "any Heist pathway" — the source-annotation standard still applies going
   forward, but this historical entry is left unannotated.

### Non-Functional Requirements

1. NFR-001: All changes MUST introduce zero new runtime dependencies. All additions are
   markdown text edits only.

2. NFR-002: Skill additions MUST be imperative instructions, not explanatory prose. No
   narrative storytelling. Additions must read as directives, not blog entries.

3. NFR-003: Per AGENTS.md, the PR for this Heist MUST include baseline pressure-test results
   (behavior WITHOUT the change) and change pressure-test results (behavior WITH the change)
   for each modified skill. All three skill changes are part of a single coherent system and
   MAY be presented as a single PR with a unified pressure-test scenario, provided that
   scenario demonstrably exercises all three surfaces: worker dispatch (resource-development
   prevention guidance), worker execution (the-hit prevention guidance), and the laundering
   blocking gate.

4. NFR-004: The laundering scan instruction MUST name the canonical identifier forms explicitly
   (e.g., FR-001, NFR-042, WP-007) so agents executing the scan know exactly what pattern to
   apply and what to look for in prevention guidance. Canonical forms include any digit count
   (FR-1, FR-42, FR-001 are all in scope). The regex `\b(FR|NFR|WP)-\d+\b` is authoritative
   over the examples where they conflict.

## Architectural Decisions

**Single hard gate at Laundering; advisory guidance upstream.** The only structural
enforcement point is the identifier scan in `skills/laundering/SKILL.md`. The additions to
`resource-development` and `the-hit` are prevention guidance — defense-in-depth that surfaces
the prohibition at dispatch time, but they are not gates. This framing is honest: workers may
still see and potentially echo identifiers from their brief context, but the scan will catch
any that survive to Laundering.

**Scope limited to confirmed leak types.** The identifier set (FR-xxx, NFR-xxx, WP-xxx) is
grounded in observed real-world leakage. Expanding to AC-xxx or AD-xxx without evidence would
violate Omerta Law 3.

**Exception mechanism with Don review.** The blocking gate includes an escape hatch for
legitimate project identifiers that coincidentally match the pattern. The agent documents
the exception; the Don reviews and acknowledges. No git-verifiable pre-existence is required —
agent judgment with Don visibility is the accepted risk level.

## Grilling Conclusions

### Key Decisions

- NC-002 scope is FR-xxx, NFR-xxx, WP-xxx only: grounded in confirmed leakage evidence; AC/AD
  excluded per Omerta Law 3 (Rule of Truth).
- Single hard gate in laundering: resource-development and the-hit additions are advisory
  (labeled honestly); scan is the sole enforcement point.
- Scan covers all project files outside `docs/gangsta/`: confirmed leaks occurred in both
  README and test fixture files.
- Exception mechanism: agent judgment + Don review is accepted risk; no git requirement.
- Source annotations are the NC standard: all NC entries carry `— Source: <heist-name>`;
  NC-001 already complies.
- Canonical forms named in prevention guidance: FR-001, NFR-042, WP-007 as examples.

### Rejected Alternatives

- Including AC-xxx and AD-xxx in NC-002: rejected — no observed leakage; would violate
  Omerta Law 3.
- Git-verifiable pre-existence for scan exceptions: rejected — too much friction for low-risk
  false-positive scenario.
- Documentation-files-only scan scope: rejected — confirmed leaks appeared in test files,
  not just markdown docs.
- No exception mechanism: rejected — WP-xxx could false-positive in some real project
  codebases (e.g. WordPress-adjacent).
- Two hard gates (the-hit report attestation + laundering scan): rejected — adds friction to
  every WP Report; Don chose single gate + advisory patches.

### Unresolved Objections

- RR-1 (accepted): Single gate fragility — if the laundering checklist item is removed or
  reordered in a future skill edit, there is no upstream redundancy. Risk: LOW. Mitigation:
  advisory guidance in the-hit and resource-development provides soft redundancy; the
  Constitution NC-002 remains binding regardless.
